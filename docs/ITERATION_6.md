### 4.3 移动止损引擎工作流

```
买入时:
  TrailingStopEngine.init_stop(entry_price)
  → 设定初始止损价 = entry_price × (1 - 8%)
  → 存储到 Position.trailing_stop (dict)

每日 _check_stop_loss 时:
  1. 读取 Position.trailing_stop
  2. TrailingStopEngine.update(state, df, danger_signals)
     a. 更新 highest_price
     b. 识别波浪谷底（N 日窗口局部低点）
     c. 止损上移至最近浪谷（只能上移）
     d. 危险信号收紧止损（收紧 30%）
     e. 最大回撤兜底（>15% 触发 EXIT）
  3. 评估信号:
     - EXIT: 触发止损价 或 跌破浪谷 3 次
     - WARNING: 跌破浪谷 2 次
     - ALERT: 跌破浪谷 1 次
     - TIGHTEN: 有危险信号
     - HOLD: 正常持有
  4. EXIT → 执行卖出 → 清除 trailing_stop 状态
```

## 6. 关键设计决策

### 6.1 规则引擎 vs Prompt 工程

**决策**：将书中可量化的交易原则编程实现为独立检查器（规则引擎），而非仅依赖 LLM prompt。

**原因**：
- LLM 可能不遵守 prompt 中的纪律要求
- 规则引擎可单元测试、可审计、不可绕过
- 规则引擎的判断结果注入 LLM prompt，形成"规则约束 + 认知增强"的双引擎模式

**代价**：
- 规则引擎的判断逻辑较粗糙（如四阶段判断依赖 MA 斜率和量比阈值）
- 需要维护规则参数（初始止损 8%、微亏容忍 2%、浪谷窗口 5 日等）
- 规则与 LLM 可能产生冲突（如规则判断"疯狂阶段不入场"但 LLM 认为"突破值得追"）

**冲突处理**：规则引擎拥有否决权（`wisdom_filtered = True`），LLM 只能在规则允许范围内调整仓位比例和止损价。

### 6.2 移动止损的浪谷识别算法

**决策**：使用 N 日窗口的局部低点作为浪谷，止损价 = 最近浪谷价格。

**算法**：
1. 在最近 `valley_window`（默认 5 日）内找最低价作为当前浪谷
2. 当价格创新高后，更新浪谷为新窗口的最低价
3. 止损价 = 最近浪谷价格（只能上移，不能下移）
4. 跌破浪谷计数：当日收盘价 < 浪谷价格时 +1

**局限性**：
- 窗口大小固定，无法适应不同波动率的股票
- 未考虑 ATR（真实波幅）自适应
- 浪谷识别可能被日内尖刺误导

### 6.3 危险信号的严重程度评估

**决策**：五级严重程度（none/low/medium/high/extreme），medium 及以上阻止买入入场。

| 信号 | 默认严重程度 | 触发条件 |
|------|-------------|----------|
| 量增价不涨 | medium | 量比 > 2 且价格涨幅 < 1% |
| 两天转头 | high | 连续 2 日收盘价下跌且累计跌幅 > 3% |
| 坏消息 | high | 外部传入（人工标记） |
| 量价背离 | medium | 价格创新高但成交量低于前 5 日均量 70% |
| 长上影线 | low | 上影线长度 > 实体 2 倍 |

### 6.4 Position 模型扩展

**决策**：在 Position dataclass 新增 `trailing_stop: dict | None` 和 `danger_signals: list[str] | None`。

**原因**：
- 移动止损状态需要每日更新并持久化
- 危险信号需要跨日追踪（连续两天转头才触发）
- 存储在 Position 中，随 PaperBroker 状态一起持久化到 JSON

## 8. 深度反思

### 8.1 从"prompt 工程"到"规则工程"的范式转换

本次迭代最重要的认知转变是：**不是所有交易智慧都适合用 prompt 工程实现**。

《炒股的智慧》中的"四阶段"、"移动止损"、"危险信号"等概念，本质上是**可量化的交易规则**，而非模糊的认知视角。将它们从 LLM prompt 中提取出来编程实现，带来了三个根本性变化：

1. **确定性**：规则引擎的判断结果是确定的、可复现的，不受 LLM 随机性影响
2. **可审计**：每个过滤决策都有明确的规则编号和参数，可追溯到具体代码行
3. **不可绕过**：规则引擎的否决权在 LLM 之前执行，确保铁律不被绕过

但这并不意味着 prompt 工程没有价值。相反，28 本书的认知视角通过 BookPerspectiveRouter 注入 LLM，在规则允许的范围内提供**认知增强**——告诉 LLM "陈江挺会怎么看这个入场机会"、"利弗莫尔会怎么设置止损"。这是规则引擎无法替代的。

**双引擎模式**：规则引擎负责"不可做什么"（否决权），LLM 认知层负责"怎么做更好"（优化建议）。

### 8.2 仓颉认知植入术的深度融合 vs 浅层集成

**浅层集成**（本次避免的）：简单地把 28 本书的 SKILL.md 全部塞进 LLM prompt，让 LLM 自己选择参考哪些书。

**深度融合**（本次实现的）：
1. **场景化路由**：不同交易场景（入场/止损/选股/风控）对应不同的书籍类别，避免信息过载
2. **核心认知提取**：从每本书的 SKILL.md 中提取"信念来源"、"绝不做的事"、"思维习惯"，而非全量注入
3. **规则化落地**：将多本书共识的可量化原则（如"禁止向下摊平"是陈江挺、利弗莫尔、海龟的共同铁律）编程实现为独立检查器

这种深度融合方式的挑战在于：需要人工判断哪些原则可量化、哪些需要 LLM 理解。未来可以考虑用 LLM 自动从 SKILL.md 中提取可量化规则候选，再由人工审核后编程实现。

### 8.3 移动止损的工程复杂度

移动止损看似简单（"随价格上涨而上移止损价"），但工程实现涉及多个复杂决策：

1. **浪谷识别**：用什么算法找浪谷？固定窗口 vs ATR 自适应 vs 分形识别？
2. **止损上移节奏**：每次上移多少？到最近浪谷？到浪谷上方一定比例？
3. **危险信号联动**：检测到危险信号时如何收紧？收紧多少？是否覆盖浪谷止损？
4. **最大回撤兜底**：即使浪谷止损未触发，回撤超过多少时强制 EXIT？
5. **状态持久化**：移动止损状态需要跨日保存，如何序列化/反序列化？

本次实现选择了较简单的方案（固定窗口 + 浪谷止损 + 危险信号收紧 30% + 15% 回撤兜底），这在大趋势跟踪中可能有效，但在震荡市中可能频繁触发假信号。需要通过回测验证不同参数组合的效果。

### 8.4 四阶段判断的精度问题

StageDetector 的判断逻辑基于以下指标：
- MA 斜率（20 日均线的变化率）
- 量比（最近 5 日均量 / 前 20 日均量）
- 波动率（20 日收益率标准差）
- 高低点结构（是否更高高高/更低低低）

这些指标在理想趋势中表现良好，但在以下场景可能误判：
- **V 型反转**：从"最后阶段"直接反转到"正常升势"，中间无过渡
- **宽幅震荡**：在牛皮市和正常升势之间频繁切换
- **缩量上涨**：量比不高但趋势明确，可能被误判为牛皮市

未来优化方向：引入 ADX（平均趋向指标）辅助判断趋势强度，或使用机器学习模型替代规则判断。

## 9. 当前不足之处

### 9.1 规则引擎参数未经验证

| 参数 | 当前值 | 风险 |
|------|--------|------|
| 初始止损比例 | 8% | 可能过宽（高频交易）或过窄（低波动蓝筹） |
| 微亏容忍阈值 | -2% | 2% 的容忍可能在高波动股票中过于严格 |
| 浪谷识别窗口 | 5 日 | 固定窗口不适应不同波动率的股票 |
| 危险信号收紧比例 | 30% | 收紧幅度可能过大，导致过早止损 |
| 最大回撤兜底 | 15% | 15% 的回撤在牛市回调中可能过于激进 |
| 四阶段 MA 斜率阈值 | 0.5% / 2.0% | 固定阈值不适应不同价格水平的股票 |

**建议**：通过回测引擎对每个参数进行网格搜索，找到最优参数组合。或引入 ATR 自适应机制。

### 9.2 移动止损缺少回测验证

移动止损引擎已集成到 TradingEngine，但**未集成到回测引擎**。这意味着：
- 无法通过历史数据验证移动止损的效果
- 无法比较"固定止损"vs"移动止损"的收益差异
- 无法优化止损参数

**建议**：在 `BacktestEngine` 中集成 TrailingStopEngine，添加 `enable_trailing_stop` 参数，支持回测对比。

### 9.3 BookPerspectiveRouter 路由规则较粗糙

当前的场景路由是基于预定义的 SCENARIO_MAPPING 字典，存在以下问题：
- 路由逻辑是静态的，不考虑当前市场环境（如熊市应更侧重风控类书籍）
- 每个场景最多返回 4 个视角，可能遗漏重要视角
- 未考虑书籍之间的认知冲突（如格雷厄姆的"安全边际"vs 利弗莫尔的"追涨"）

**建议**：
1. 引入市场状态感知的路由（牛市偏向成长/技术分析，熊市偏向价值/风控）
2. 增加认知冲突检测，在 prompt 中标注"以下视角存在分歧"
3. 用 LLM 动态选择视角（给定场景和当前信号，让 LLM 选择最相关的 3 本书）

### 9.4 Web 看板缺少交互能力

当前 Web 看板是纯展示性的，无法：
- 手动确认/拒绝信号
- 调整止损价
- 查看信号详情（点击展开完整决策链路）
- 查看书籍视角的完整 SKILL.md 内容

**建议**：在阶段四前，为 Web 看板增加信号确认、止损调整等交互功能。

### 9.5 危险信号的"坏消息"检测依赖人工

五大危险信号中，"坏消息"信号需要外部传入（人工标记），系统无法自动检测。其他四个信号虽然自动检测，但判断逻辑较简单：
- "量增价不涨"仅看量比和涨幅，未考虑大单 vs 小单
- "两天转头"未区分正常回调 vs 趋势反转
- "量价背离"的 70% 阈值是经验值，未经验证
- "长上影线"的 2 倍实体阈值可能不适用所有股票

**建议**：
1. 接入新闻 API（如东方财富公告）自动检测坏消息
2. 引入大单数据区分"散户放量"vs"主力放量"
3. 用 ATR 标准化上影线长度

### 9.6 缺少 P1/P2 级别优化

本次迭代仅完成了 P0 级别的 4 项核心功能。Review 报告中提出的 P1/P2 级别优化尚未实施：

| 优先级 | 项目 | 说明 |
|--------|------|------|
| P1 | 多周期共振入场 | 日线 + 周线 + 月线趋势一致才入场 |
| P1 | 板块轮动模型 | 资金在板块间的流动规律 |
| P1 | 动态仓位管理 | 基于 Kelly 公式或波动率调整仓位 |
| P2 | 回测引擎集成认知引擎 | 回测时启用四阶段/移动止损/危险信号 |
| P2 | 净值曲线记录 | 每日快照记录，支持净值曲线绘制 |
| P2 | JSON 日志格式 | 便于监控系统解析 |
| P3 | 飞书命令回执 | 用户通过飞书确认/拒绝交易 |

## 10. 后续优化路线图
