# 迭代 1：全量专家 Review 与基础治理

## 1. 业务理解

**gugu** 是一个面向 A 股的自动化交易系统，解决的核心痛点是：

- 散户/个人投资者没有时间和精力实时盯盘、执行交易纪律；
- 人工选股容易被情绪左右，缺乏统一的风控与回验；
- 交易信号、持仓、盈亏信息分散，无法及时获知并决策。

系统期望通过“策略固化 → 回测 → 模拟盘 → 小额实盘 → 正式实盘”五阶段，逐步把“选股-买卖-风控-通知”全流程自动化，并用飞书把关键信息实时推送给用户。

## 2. 本次迭代目标

- 对当前完整代码库和文档做一次系统性“专家体检”；
- 从架构、产品、开发、用户体验、运营、数据、测试七个维度找出可修复的问题；
- 全部修复后通过 `ruff / mypy / pytest` 三重质量门禁；
- 输出/完善产品文档（PRODUCT.md）和技术文档（TECH.md），让非技术用户和后续开发者都能快速理解系统。

## 3. 范围

覆盖文件：

- `CLAUDE.md`（项目公约与架构）
- `pyproject.toml`、`config/*.yaml`
- `src/gugu/**/*.py`
- `tests/**/*.py`
- `scripts/*.py`

不范围：

- 引入新的交易策略或数据源；
- 接入真实 QMT/飞书 API（保留模拟与可配置）；
- 重大架构重构（仅做必要的小幅治理）。

## 4. 验收标准

- [ ] 无 `ruff` 告警；
- [ ] 无 `mypy` 错误；
- [ ] `pytest` 全绿且覆盖率 ≥ 70%；
- [ ] 产品文档与技术文档补充到能独立说明“系统是什么、解决什么问题、怎么用、怎么扩展”；
- [ ] 第二次全量 expert review 没有新增 P1/P2 问题。

## 5. 专家 Review 维度

| 专家 | 关注重点 |
|------|----------|
| 架构专家 | 模块边界、依赖方向、可扩展性、五阶段演进是否可落地 |
| 产品专家 | 用户痛点是否被准确解决、功能闭环、MVP 边界 |
| 开发专家 | 代码规范、类型安全、异常处理、可维护性 |
| 用户体验专家 | CLI/脚本入口、配置体验、飞书消息可读性、错误提示 |
| 运营专家 | 日志、监控、日报/告警、风控熔断后的恢复机制 |
| 数据专家 | 数据质量、缓存、降级、交易日历、回测前视偏差 |
| 测试专家 | 单元测试覆盖、集成测试缺口、fixture 可复用性 |
