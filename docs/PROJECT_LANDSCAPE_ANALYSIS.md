# 项目全景分析：MakingMoney / MingCe / realtime-flow / gugu

> 基于第一性原理和对抗性审查，2026-06-29

---

## 一、第一性原理：用户真正的需求是什么？

### 1.1 本质问题

作为个人 A 股投资者，最底层的需求只有一个：

> **"在市场中持续做出正确的交易决策，并严格执行。"**

围绕这个本质，衍生出四个基础能力：

| 能力 | 描述 | 不能没有的原因 |
|------|------|----------------|
| **数据** | 获取准确、及时的行情数据 | 没有数据，一切决策都是赌博 |
| **信号** | 从数据中提取可操作的买卖信号 | 手工看 5000 只股票不现实 |
| **风控** | 控制每笔交易的风险暴露 | 一次大亏可能抹掉十次小赚 |
| **认知** | 理解市场状态，避免情绪化决策 | 纯量化信号在极端行情下全失效 |

### 1.2 现有项目的演化轨迹

```
MakingMoney (v1.0)       ← 全栈量化平台，27 策略，Flask UI
     │
     ├──→ MingCe (v1.5)  ← 提取日报 Bot，5 时段飞书推送，五维宏观框架
     │
     ├──→ realtime-flow  ← 资金流监测专用，WebSocket 推送
     │
     └──→ gugu (v2.0)    ← 重构版：信号监控 + P0 认知引擎 + 28 本书
```

这不是三个独立系统——这是一个系统的三次迭代。

## 二、对抗性审查：质疑每个决策

### 审查 1：需要维护多个代码库吗？

**主张**：MakingMoney、MingCe、gugu 三个项目皆独立维护。

**对抗性质疑**：
- MakingMoney 已事实废弃（无 README，pyproject.toml 引用不存在的文件）
- MingCe 的 `engine/` 指向的是 MakingMoney 的代码拷贝——同一份策略逻辑在两个地方存在
- gugu 已经包含了 MakingMoney 的核心能力（策略、风控、通知），并且多了测试和 P0 认知引擎
- 每次修改策略参数需要在 2-3 个地方同步改

**结论**：❌ 不应该维护多个。但 MingCe 是当前生产运行的日报系统，**不能直接删除**。

### 审查 2：MingCe 和 gugu 的功能重叠度有多高？

**主张**：MingCe 做日报，gugu 做信号监控，二者互补。

**对抗性质疑**：

| 维度 | MingCe | gugu |
|------|--------|------|
| 数据源 | akshare + 多级降级 | akshare + sina 降级 |
| 策略 | 18 个（6 类） | 8 个（3 类） |
| 风控 | 三级风控 | 三级风控 |
| 飞书通知 | 5 时段卡片推送 | 信号 + 日报 + 风控告警 |
| LLM 分析 | 三层传导 + 大师兄 | WisdomAdvisor + LLM 决策 |
| 知识库 | 《炒股的智慧》7 skill | 28 本书籍多视角 |
| 测试 | 无 | **506 个测试，100% 通过** |
| 架构 | FastAPI + 脚本 | 清洁包结构，Pydantic 配置 |
| 运行 | launchd 保活 | launchd 待配置 |

**关键发现**：
- gugu 在 **架构质量** 上远超 MingCe（测试、模块化、类型安全）
- MingCe 在 **日报功能** 上更成熟（5 时段推送、五维框架、基金跟踪）
- 策略重叠：MingCe 有 18 个（含 LLM 生成策略），gugu 有 8 个（精简过的前 3+）——gugu 的 settings.yaml 只启用了 3 个（box_breakout、turtle、dual_ma），因为回测显示其他策略亏损

**结论**：✅ 高重叠，**应合并而非并行维护**。

### 审查 3：合并的风险和收益

**主张**：合并会破坏 MingCe 的日报生产流程。

**对抗性质疑**：

| 风险 | 概率 | 缓解措施 |
|------|------|----------|
| 日报推送中断 | 中 | 双轨运行：MingCe 日报不停止，gugu 逐步接管 |
| 策略信号差异 | 低 | 信号差异本身可做 A/B 测试 |
| 五维框架丢失 | 中 | gugu 缺少宏观看板→需补 |
| 用户适应成本 | 低 | 飞书卡片样式可保持一致 |

**收益**：
- 消除 3 个代码库的维护开销
- 策略逻辑统一在一个地方
- 测试覆盖 MingCe 的日报功能
- 未来功能迭代只需改一处

**结论**：✅ 收益远大于风险。

### 审查 4：realtime-flow 还有存在价值吗？

**主张**：realtime-flow 的资金流分析模型值得保留。

**对抗性质疑**：
- realtime-flow 的核心逻辑（主力净占比评分、板块轮动）已被 gugu 的 `src/gugu/analysis/` 和 `src/gugu/filters/` 吸收
- realtime-flow 的 WebSocket Web 界面是独立的（端口 8899），与 gugu 无关
- 原始 realtime-flow 项目（`~/workspace/realtime-flow/`）已经只有一个 `db/` 目录——代码被删了

**结论**：✅ realtime-flow 的参考价值已完全被 gugu 吸收。`_refs/realtime-flow/` 可保留作为文档参考，独立项目可归档。

## 三、综合决策

### 3.1 战略结论

> **将 MingCe 的核心功能逐步迁移到 gugu 中，然后归档 MakingMoney、MingCe、realtime-flow 三个独立项目。未来只有 gugu 一个项目。**

### 3.2 迁移路线图

| 阶段 | 内容 | 优先级 |
|------|------|--------|
| **P0** | 补齐 gugu 缺少的 MingCe 功能 | 立即 |
| P0.1 | 五维宏观框架（金油汇债G） | 关键——gugu 目前只有股票信号，缺少宏观维度 |
| P0.2 | 基金净值跟踪卡片 | 用户已有此需求（settings.yaml 中有基金推送时段） |
| P0.3 | 多时段日报模板（早间简报/午间/收盘/基金） | gugu 目前只有 scan 信号推送 |
| **P1** | 补齐 gugu 的策略覆盖 | 高 |
| P1.1 | 迁移 MingCe 的回测验证过的策略到 gugu（18→8 已有，需评估是否加回） | settings.yaml 已精简，按回测决策 |
| P1.2 | 四维数据质量评分系统 | MingCe 有完整的数据质量框架 |
| **P2** | 生产运行 | 中 |
| P2.1 | gugu 调度器替换 MingCe 的日报调度 | 停止 MingCe 的端口 8000/8765 |
| P2.2 | launchd 保活 gugu 调度器 | 替换 com.market-daily-bot |
| **P3** | 清理 | 低 |
| P3.1 | MakingMoney 代码冻结，标注 DEPRECATED | 保留只读 |
| P3.2 | MingCe 代码冻结，标注 MERGED INTO GUGU | 保留只读 |
| P3.3 | realtime-flow 归档 | 移出 workspace |

### 3.3 各项目最终状态

| 项目 | 建议状态 | 原因 |
|------|---------|------|
| **MakingMoney** | 🟡 冻结 | 历史 v1.0 原型，代码可查阅但不修改 |
| **MingCe** | 🟡 冻结→逐步退役 | 当前日报系统，等 gugu 接管后归档 |
| **realtime-flow** | 🔴 归档 | 资金流分析已被 gugu 吸收 |
| **gugu** | 🟢 唯一活跃 | 未来所有迭代都在此进行 |

## 四、gugu 当前需要补齐的 MingCe 功能清单

基于第一性原理——如果 gugu 是唯一的系统，它必须覆盖用户的所有需求。以下是从 MingCe 中需要迁移的：

### 4.1 五维宏观数据看板（金油汇债G）

MingCe 有一个成熟的五维框架，每天早上推送全球资产简报。这是用户做个股决策的上层参考，不应该丢失。

需要新增模块：
- `src/gugu/macro/` — 宏观数据采集（伦敦金、布伦特/WTI 原油、美元指数、USDCNY、美债 10Y/2Y 收益率、VIX、BDI）
- `src/gugu/macro/report.py` — 宏观报告生成模板
- `src/gugu/notifier/macro_card.py` — 宏观飞书卡片渲染

配置新增：
- `macro.enabled` / `macro.primary_source` / `macro.fallback_sources`
- 推送时段：08:00（全球隔夜简报）

### 4.2 多时段日报流程

MingCe 的日报推送是目前用户每天实际在使用的功能：

| 时段 | 时间 | gugu 现状 | 需要补齐 |
|------|------|-----------|---------|
| 全球早间简报 | 08:00 | ❌ 无 | 五维全球资产概览 |
| A 股盘前 | 09:10 | ⚠️ 有配置但未实现 | 板块热点 + 隔夜影响 + 昨日回顾 |
| 午间 | 11:35 | ⚠️ 有配置但未实现 | 上午盘面 + 板块轮动 |
| 收盘 | 15:10 | ⚠️ 有配置但未实现 | 全天数据 + 策略信号 + 认知分析 |
| 基金 | 15:35 | ❌ 无 | 基金净值跟踪 |

### 4.3 飞书交互式指令

MingCe 支持 @机器人指令：
- `关注/取消关注 股票代码` — 添加到自选
- `持仓/移除持仓 代码 数量` — 手动更新持仓
- `信号查询` — 当前策略信号
- `深度分析 代码` — LLM 驱动深度分析
- `查看组合` — 当前持仓概览

gugu 的飞书通知只有单向推送，没有交互指令。

## 五、第一性原理结论

> **维护 3 个功能重叠的代码库，本质上是在维护"遗忘"——忘记了之前的修改，忘记了哪个版本有哪个功能，忘记了 Bug 是在哪个库修复的。**
>
> 从第一性原理出发：用户只需要"一个系统"，能完成"数据→信号→风控→认知→执行"的完整闭环。
>
> 当前的最优解是：**停止扩张，集中到 gugu，逐步吸收 MingCe 的成熟功能。**
